The Problem

Devnets are complex.
Debugging shouldn't be.

When something goes wrong in a multi-client Ethereum devnet, you need answers — not raw metric dumps. Existing dashboards don't understand Kurtosis topologies, multi-participant labeling, or per-client internals.

🔍

No per-client debug dashboards

Each EL and CL client exposes unique native metrics. No open-source dashboard collection covers all major clients with Kurtosis-aware variables and multi-participant filtering.

🔀

Different metric names, same concepts

Geth uses chain_head_block. Reth uses reth_blockchain_tree_canonical_chain_height. Nethermind differs again. Cross-client comparison requires careful normalization.

🏷️

No devnet-aware labeling

Home validator dashboards hardcode targets. Kurtosis devnets use dynamic service names. Helix extracts participant labels automatically via Prometheus relabel configs.

01

Client-Native Metrics

Each Ethereum client exposes its own internals: database latency, sync stages, Engine API timings, runtime pressure. Helix curates the most signal-dense metrics per client.

Explain monitoring tools, document metric schemas, and build setup guides. Great for beginners.

Run clients, capture /metrics, and map which metrics are stable and meaningful across versions.

⚡

Prometheus Config

Add scrape jobs, normalize labels with relabel_configs, write recording and alerting rules with unit tests.

🎨

Grafana Dashboards

Design panels, build dashboard variables, and create new per-client or topic-based visualizations.

🧪

Validation & CI

Write dashboard validation scripts, promtool unit tests, and provisioning checks to keep everything reliable.

🔌

New Clients

Add support for Grandine and other emerging Ethereum clients as their metric APIs stabilize.
